# F1: Accessibility Tree (AXTree) for Web Scraping

## Findings

### [1] Playwright MCP's snapshot format is a YAML-like indented tree where each line encodes role, accessible name, and a unique ref for interactive elements
- quote: "- heading \"todos\" [level=1] — textbox \"What needs to be done?\" [ref=e5] — listitem: — checkbox \"Toggle Todo\" [ref=e10]"
- url: https://playwright.dev/mcp/snapshots
- source_type: primary
- published: 2026
- confidence: high

### [2] Refs use format `e` + number (e1, e15, e203), are stable within a single snapshot, and are only assigned to interactive elements (buttons, links, inputs)
- quote: "Only interactive elements get refs (buttons, links, inputs, etc.) — Valid until the next page change — Unique within a single snapshot"
- url: https://playwright.dev/mcp/snapshots
- source_type: primary
- published: 2026
- confidence: high

### [3] Playwright MCP snapshots consume ~200-400 tokens per page vs ~3000-5000 for screenshots or full DOM, and the snapshot is deterministic given a static page
- quote: "Token cost: ~200-400 tokens vs ~3000-5000 tokens (vision model) — Precision: Exact — refs point to specific elements — Reliability: Deterministic"
- url: https://playwright.dev/mcp/snapshots
- source_type: primary
- published: 2026
- confidence: high

### [4] The snapshot carries role-specific properties in bracket annotations: checkbox gets [checked=true], heading gets [level=1], textbox gets [value=...], etc.
- quote: "Properties surface as bracket-suffix annotations: checkbox \"I accept\" [checked=true]"
- url: https://qaskills.sh/blog/playwright-mcp-accessibility-snapshots-reference
- source_type: secondary
- published: May 3, 2026
- confidence: high

### [5] Playwright MCP assigns refs to ALL elements including non-interactive ones (generic, rowgroup, cell), producing ~789 refs on GitHub vs WebClaw's 245 which only refs interactive elements
- quote: "Playwright MCP — labeling every element with a ref — gives maximum flexibility for targeting any element. WebClaw trades that completeness for compactness by only labeling things the AI can actually interact with."
- url: https://dev.to/kuroko1t/how-accessibility-tree-formatting-affects-token-cost-in-browser-mcps-n2a
- source_type: secondary
- published: February 26, 2026
- confidence: high

### [6] Format-only differences between AXTree tools produce 51-79% token savings on real pages; GitHub page dropped from 19,409 tokens (Playwright MCP) to 4,304 tokens (WebClaw)
- quote: "GitHub (anthropics/claude-cookbooks): 19,409 tokens (Playwright MCP) → 4,304 tokens (WebClaw) — 78% smaller"
- url: https://dev.to/kuroko1t/how-accessibility-tree-formatting-affects-token-cost-in-browser-mcps-n2a
- source_type: secondary
- published: February 26, 2026
- confidence: high

### [7] agent-browser is a 100% native Rust CLI (no Playwright/Puppeteer dependency) that produces accessibility tree snapshots with compact `@eN` ref format, designed for ~200-400 token context efficiency
- quote: "100% native Rust — Chrome/Chromium via CDP with no Playwright or Puppeteer dependency — Accessibility-tree snapshots with element refs for reliable interaction"
- url: https://agent-browser.dev/
- source_type: primary
- published: 2026
- confidence: high

### [8] agent-browser uses `@eN` prefix (e.g. `@e1`, `@e2`) for refs, differs from Playwright MCP's bare `eN` format
- quote: "agent-browser click @e2 — Snapshot returns accessibility tree with refs"
- url: https://agent-browser.dev/
- source_type: primary
- published: 2026
- confidence: high

### [9] Cloudflare Browser Run exposes a `/accessibilityTree` REST endpoint that returns JSON with nested `{role, name, children}` nodes — no refs, just structural data
- quote: "{ \"role\": \"RootWebArea\", \"name\": \"Example Domain\", \"children\": [{ \"role\": \"heading\", \"name\": \"Example Domain\", \"level\": 1 }, ...] }"
- url: https://developers.cloudflare.com/browser-run/quick-actions/accessibility-tree-endpoint
- source_type: primary
- published: July 7, 2026
- confidence: high

### [10] CDP's `Accessibility.getFullAXTree` returns a flat array of nodes with typed objects (`{type: "role", value: "heading"}`), not flat strings — Lightpanda and Chrome both implement this
- quote: "role: { type: \"role\", value: \"<role>\" } — name: { type: \"computedString\", value: \"<name>\", sources: [...] } — properties: array of ARIA properties"
- url: https://lightpanda.io/docs/guides/markdown-axtree
- source_type: primary
- published: 2026
- confidence: high

### [11] Lightpanda provides a `semantic_tree` CLI format with flat JSON fields (role, name, isInteractive, xpath) distinct from CDP's typed-object schema — the two are not interchangeable
- quote: "The CLI semantic_tree and the CDP Accessibility.getFullAXTree return different JSON schemas. The CLI uses flat field names (role, name as strings); CDP uses typed objects."
- url: https://lightpanda.io/docs/guides/markdown-axtree
- source_type: primary
- published: 2026
- confidence: high

### [12] The agent-browser-loop pattern for scraping is: snapshot → LLM parses refs → issue action by ref → re-snapshot → verify; stale refs must never be reused
- quote: "Refs are scoped to the snapshot in which they were created. After any action, take a new snapshot before issuing the next interaction."
- url: https://qaskills.sh/blog/playwright-mcp-accessibility-snapshots-reference
- source_type: secondary
- published: May 3, 2026
- confidence: high

## Dead ends
- The arxiv paper "Building Browser Agents" (2511.19477) mentions AXTree briefly but only as one tool among many in agent architecture surveys — no concrete data format details.
- Searching for "AXTree JSON schema specification W3C" returns the W3C accessibility API mappings (core-aam) but not a standalone JSON schema — there is no single JSON spec; each tool (CDP, Playwright, Cloudflare) defines its own shape.

## Suggested follow-ups
- How does Playwright's internal `accessibility.snapshot()` (the `interestingOnly: false` mode) map to the MCP snapshot format — what filtering/transformation happens between the raw CDP AXTree and the YAML-like MCP output?
- What is the actual token cost of agent-browser's `@eN` snapshot on large pages (500+ interactive elements) vs Playwright MCP — does the compact format hold at scale?
- For a Rust scraper (webfang), would it be feasible to use CDP's `Accessibility.getFullAXTree` directly and apply our own compact serialization, bypassing both Playwright and agent-browser?
