# F2: LLM-First Extraction (Schema + Markdown)

## Findings

### [1] Crawl4AI's `LLMExtractionStrategy` is provider-agnostic via LiteLLM and always runs LLM calls server-side
- quote: "Crawl4AI uses a 'provider string' (e.g., 'openai/gpt-4o', 'ollama/llama2', 'aws/titan') to identify your LLM. Any model that LiteLLM supports is fair game."
- url: https://docs.crawl4ai.com/extraction/llm-strategies
- source_type: primary
- published: 2025-01-01
- confidence: high

### [2] Crawl4AI's LLM extraction accepts Pydantic JSON schemas and returns validated JSON via `extraction_type="schema"`
- quote: "You define a schema (like a Pydantic model) or a simpler 'block' extraction approach... For structured data, 'schema' is recommended. You provide schema=YourPydanticModel.model_json_schema()."
- url: https://docs.crawl4ai.com/extraction/llm-strategies
- source_type: primary
- published: 2025-01-01
- confidence: high

### [3] Crawl4AI's extraction API has two key methods: `extract(url, html)` and `run(url, sections)` — the orchestrator agent calls these, and the LLM call happens server-side within the strategy
- quote: "All extraction strategies inherit from the base ExtractionStrategy class and implement two key methods: extract(url: str, html: str) -> List[Dict[str, Any]] and run(url: str, sections: List[str]) -> List[Dict[str, Any]]"
- url: https://docs.crawl4ai.com/api/strategies
- source_type: primary
- published: 2025-01-01
- confidence: high

### [4] Crawl4AI offers a one-time LLM-assisted schema generation utility that produces reusable CSS/XPath schemas, eliminating ongoing LLM costs
- quote: "While schema generation uses LLM, it's a one-time cost. The generated schema can be reused for unlimited extractions without further LLM calls."
- url: https://docs.crawl4ai.com/extraction/no-llm-strategies
- source_type: primary
- published: 2025-05-02
- confidence: high

### [5] Firecrawl's `/scrape` endpoint accepts Pydantic schemas via `formats` list with `{"type": "json", "schema": ...}` and returns typed JSON alongside markdown in the same response
- quote: "The formats list takes strings and dicts together. 'markdown' gets the raw page text, and the JSON format dict tells the API to run LLM extraction against your schema. Both come back in the same response."
- url: https://www.firecrawl.dev/blog/ai-powered-data-retrieval
- source_type: primary
- published: 2026-03-03
- confidence: high

### [6] Firecrawl's `/extract` endpoint supports both `prompt` (natural language) and `schema` (rigid JSON structure) as mutually optional parameters, with wildcard URL support for full-domain extraction
- quote: "Provide a list of URLs, optionally with wildcards (e.g., example.com/*), and a prompt or schema describing the information you want. Firecrawl handles the details of crawling, parsing, and collating large or small datasets."
- url: https://docs.firecrawl.dev/features/extract
- source_type: primary
- published: unknown
- confidence: high

### [7] Firecrawl's `/agent` endpoint takes a natural language prompt + Pydantic schema directly (no `.model_json_schema()` needed) and uses its own models (spark-1-mini/spark-1-pro) to autonomously browse, read, and extract
- quote: "Unlike scrape's JSON format, the agent endpoint accepts a Pydantic class directly as the schema argument. No .model_json_schema() call needed... Two model choices: spark-1-mini (default) costs less and works well for straightforward questions. spark-1-pro is better for complex or ambiguous research."
- url: https://www.firecrawl.dev/blog/ai-powered-data-retrieval
- source_type: primary
- published: 2026-03-03
- confidence: high

### [8] Firecrawl's MCP server exposes 12 tools via Model Context Protocol, including all extraction endpoints, making it natively callable by MCP agent servers
- quote: "Run npx firecrawl-mcp with your API key, and you get 12 tools exposed through the Model Context Protocol: the four endpoints from this tutorial, plus map, batch scrape, browser automation, and async job management."
- url: https://www.firecrawl.dev/blog/ai-powered-data-retrieval
- source_type: primary
- published: 2026-03-03
- confidence: high

### [9] Client-side SLM extraction is now viable with sub-1GB quantized models (e.g. SmolLM2-360M) running via WebGPU/ONNX in the browser, achieving ~70-85% valid JSON on single-entity schemas without post-processing
- quote: "Specialized small language models, quantized to run under 1GB, can now perform inference directly in the browser via WebGPU and ONNX Runtime Web... Without post-processing, roughly 70%–85% of outputs parsed as valid JSON in our limited tests on single-entity schemas."
- url: https://www.sitepoint.com/slm-structured-data-extraction-browser
- source_type: secondary
- published: 2026-02-25
- confidence: medium

### [10] The dominant architecture for LLM-first extraction in 2026 is server-side: the scraping service fetches + cleans HTML, sends it to an LLM with a user-defined schema, and returns structured JSON — the orchestrator agent never touches the raw HTML
- quote: "All the fetching and formatting happens on the API side, which is what makes these APIs a good fit for products that need web data without the overhead of maintaining a scraping pipeline."
- url: https://www.firecrawl.dev/blog/ai-powered-data-retrieval
- source_type: primary
- published: 2026-03-03
- confidence: high

### [11] Crawl4AI's `LLMExtractionStrategy` supports automatic chunking with overlap for large pages, sending chunks to the LLM in parallel and merging results — critical for token-limit management
- quote: "chunk_token_threshold sets the approximate max tokens per chunk... overlap_rate=0.1 means each subsequent chunk includes 10% of the previous chunk's text... you can potentially process multiple chunks in parallel."
- url: https://docs.crawl4ai.com/extraction/llm-strategies
- source_type: primary
- published: 2025-01-01
- confidence: high

### [12] Schema quality, not schema support, is the failure mode at scale: GPT-4 shows 11.97% invalid response rate on complex extraction tasks, and Amazon's PARSE research lifted valid JSON rates from 82.3% to 98.7% via schema refinement
- quote: "Amazon's PARSE research traces the problem to a mismatch: JSON schemas were built as contracts between human developers and static systems, not as instructions for LLMs... GPT-4 shows an 11.97% invalid response rate on complex extraction tasks."
- url: https://www.context.dev/blog/best-structured-data-extraction-apis-for-llms-2026
- source_type: secondary
- published: 2026-06-27
- confidence: medium
